# NNUE 特徴量設計（変成将棋版 HalfKP + キングバケット）

## 概要

キングバケット方式を採用する。  
自玉・敵玉の獅王状態（3通り）ごとに専用の W1 重みを持つことで、  
特徴量から is獅子 次元を排除しつつ局面の性質の違いを表現する。

> 自玉=獅王 かつ 敵玉=獅王 はルール上発生不可能。バケットは3通り。

| バケット | 自玉 | 敵玉 |
|---------|------|------|
| 0 | 玉将 | 玉将 |
| 1 | 玉将 | 獅王 |
| 2 | 獅王 | 玉将 |

**特徴空間**: 196,425 × 2視点 = 392,850  
**W1 重み**: 392,850 × 256 × 3バケット = 合計 302M パラメータ（int8 で約 **289MB**）  
**推論時の活性 W1**: 1バケット分 = 約 **48MB**（int8）

## 動作フロー

```
1. 現局面の (is自玉獅子, is敵玉獅子) → バケット番号 (0〜3) を決定
2. 対応する W1[bucket] を使ってアキュムレータを計算
3. 玉が成って獅王になったとき → バケット変更 → アキュムレータ全再計算
4. 通常の手 → インクリメンタル更新（2〜4行だけ差分）
```

## 備考

- **キングバケット採用理由**: 玉将と獅王では逃げ筋・詰み筋が全く異なるため、局面フェーズごとに専用の重みで学習させる
- **推論時の活性 W1**: 3バケット中 1つだけ使用 → 実効 48MB（int8）→ キャッシュ効率が3倍改善
- **バケット切り替えコスト**: 玉の成りは1局に1〜2回程度のため再計算コストは許容範囲
- **スパース性**: アクティブな特徴数は約 30〜40 個（盤上の駒数 + 持駒段階数）

## 設計上の考察

### バケット方式 vs 単一W1（is獅子次元込み）の比較

当初、キングバケットの利点として「バケットごとに W1 が小さくなりキャッシュ効率が上がる」を挙げたが、これは**成立しない**。

- αβ探索中にアクティブになる W1 の行は約40行 × 256byte = **10KB**
- この10KBは W1 全体が47MBであれ2.6MBであれ、**L1キャッシュ（32KB）に収まる**
- 同じ局面変化を何千ノードも読む探索の性質上、同じ行が繰り返し参照されキャッシュに残る

よって**バケット方式の実際のメリットは「フェーズ別の重み専門化（学習品質）」のみ**であり、キャッシュ効率の改善は期待できない。

### 獅子成り時の全再計算について

is自玉獅子 が変わると、自玉位置を文脈として持つ**全特徴量のインデックスが変わる**。これはバケット方式でも単一W1方式でも同様であり、成り時の全再計算は設計上避けられない。

ただし、本家 HalfKP と比較すると：

| | 全再計算のタイミング | 頻度 |
|--|--|--|
| 本家 HalfKP | 玉が動くたびに必要 | 1局に数十回 |
| **この設計** | **獅子に成るときのみ** | **1局に1〜2回** |

獅子に成ることの局面的な意味が大きいことを考えると、コストに見合った設計と言える。

### αβ探索木でのバケット切り替え

成りが絡む局面（両玉が中段で成りを争う状況など）では、探索木の中でバケット0→1→0→2… と切り替わりうる。1局の対局中は1〜2回でも、αβ探索のノード数基準では無数に発生する。これはインクリメンタル更新の恩恵が減る局面でもある。

ただし、こうした局面は終盤の重要局面であることが多く、多少の速度低下は許容範囲と判断する。

### 成り後のバケット1・2間の「行き来」について

HalfKP では先手視点（acc[0]）と後手視点（acc[1]）の **2つのアキュムレータを毎手セットで更新** する。一方が動かした駒は両方の玉から見て位置が変わるため、片方だけ更新することはできない。

先手玉が獅王に成った後は：
- acc[0]（先手視点）→ 常にバケット2の W1 で差分更新
- acc[1]（後手視点）→ 常にバケット1の W1 で差分更新

「バケット1と2を行き来する」と表現したが、実際には **毎手両方を同時に更新しているだけ** であり、全再計算が発生するのは成りの瞬間一度だけである。成り後は2つのアキュムレータが各自のバケットで安定してインクリメンタル更新され続けるため、処理上の大きな差異はない。
