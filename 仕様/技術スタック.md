# 変成将棋 技術スタック方針

## 構成

| レイヤー | 言語・技術 | 役割 |
|---------|-----------|------|
| (1) UI | C# / WPF | 盤面表示・駒操作 |
| (2) ゲームエンジン | C# (`変成将棋.Engine`) | 合法手生成・局面管理・千日手判定 |
| (3) AI 推論（対局） | C# (`変成将棋.AI`) + ONNX Runtime | MCTS + ニューラルネット推論 |
| (4) 学習器 | Python / PyTorch | 自己対局・ニューラルネット学習（AlphaZero方式） |
| (5) C# ↔ Python 橋 | NativeAOT + ctypes | 学習時のゲームエンジン呼び出し |

## 開発順序

(1) → (2) → (4) → (3) の順に実装する。  
alpha-beta 探索は作らない。

## AI アーキテクチャ（AlphaZero 方式）

### ニューラルネット構造（固定・変更不可）

- **入力**: (47, 9, 9) のテンソル
  - ch  0-15: 先手の各駒種（E駒種 1-16）
  - ch 16-31: 後手の各駒種
  - ch 32-38: 先手持ち駒枚数（7種、全升にブロードキャスト）
  - ch 39-45: 後手持ち駒枚数
  - ch 46:    現在手番（先手=1）
- **バックボーン**: ResNet 10ブロック / 128チャンネル（約 610万パラメータ）
- **出力ヘッド**:
  - Policy head: 18,873 次元のロジット（合法手インデックス空間）
  - Value head: -1〜+1（現局面の勝率）

### ポリシーインデックス空間（合計 18,873）

| 範囲 | 手の種類 | 計算式 |
|------|---------|--------|
| 0〜13,121 | 盤上移動 | from(81) × to(81) × promote(2) |
| 13,122〜13,688 | 駒打ち | piece(7) × to(81) |
| 13,689〜18,872 | 獅王2回移動 | from(81) × via方向(8) × to方向(8) |

### MCTS: Gumbel AlphaZero

標準 PUCT の代わりに Gumbel AlphaZero を採用（"Policy improvement by planning with Gumbel", DeepMind 2022）。

- **Gumbel ノイズ**: ルートノードの探索に Gumbel-max trick を使用
- **Completed Q-value**: 未訪問ノードの価値を訪問済み平均で補完
- **学習ターゲット**: 訪問数ではなく「改善ポリシー」（log π + C × σ(Q_comp)）
- **効果**: 32〜64 sims で 200 sims PUCT と同等以上の品質

### バッチ推論

複数の MCTS リーフを溜めてから 1 回の GPU フォワードパスで評価する：

```
通常（逐次）: sim1→NN, sim2→NN, ...  GPU使用率 <5%
バッチ:       [sim1, sim2, ..., sim8]→NN（1回）GPU使用率 向上
```

Virtual loss で並列シミュレーションの経路重複を防ぐ。

## C# ↔ Python 連携方針

### 学習時（NativeAOT + ctypes）

```
Python MCTS → ctypes → shogi_engine.dll/.so（NativeAOT）
Python PyTorch → GPU → NN 推論
```

NativeAOT は C# を `shogi_engine.so`（Linux）/ `shogi_engine.dll`（Windows）に
AOT コンパイルし、Python が ctypes で直接呼ぶ。

| 方法 | 橋のオーバーヘッド | GPU での速度 | CPU での速度 |
|------|-----------------|------------|------------|
| Python.NET（旧） | ~8ms/call × 3 = 24ms | ~10ms/sim | ~53ms/sim |
| NativeAOT + ctypes | ~0.1ms/call × 1 = 0.1ms | **~2ms/sim** | ~43ms/sim |

`shogi_expand` バッチ API（apply + tensor + legal_moves を 1 呼び出しで）で
ブリッジ呼び出し回数を 3 回 → 1 回に削減。

**GPU では 5〜10 倍、CPU では 1.4 倍の高速化**（CPU は NN 推論が支配的なため）。

### 推論時（対局）

```
WPF UI → CMCTSPlayer（C#）→ C# Engine（直接、橋なし）
                           → OnnxRuntime → ONNX モデル
```

対局時は C# → C# のネイティブ呼び出しのため橋のオーバーヘッドはゼロ。

### NativeAOT ビルド方法

**Windows:**
```powershell
$env:PATH = "C:\Program Files (x86)\Microsoft Visual Studio\Installer;$env:PATH"
dotnet publish 変成将棋.Engine.Native -r win-x64 -c Release --self-contained
```

**Linux (Colab):**
```bash
apt-get install -y clang zlib1g-dev
dotnet publish 変成将棋.Engine.Native -r linux-x64 -c Release --self-contained
```

## AlphaZero 以降の改良について

MiniZero（IEEE ToG）で各手法がモジュール独立で組み合わせ可能であることが実証されており、
以下の組み合わせに食い合わせはないことを確認した。

| 手法 | 概要 | 採用状況 |
|------|------|---------|
| **Gumbel AlphaZero（2022）** | 少ないシミュレーションで高品質な探索 | ✅ **採用** |
| **KataGo 方式（補助タスク）** | ownership/score 予測で収束加速・Global Pooling | ✅ **採用** |
| **優先度付き経験再生（PER）** | TD 誤差に基づく優先サンプリング | ✅ **採用** |
| **Value Prefix 予測** | N 手先ブートストラップで価値ターゲットの分散削減 | ✅ **採用** |
| **MuZero（2020）** | ルールも学習する環境モデル（設計変更が大きい） | ❌ 未採用 |
| **NNUE（将棋/チェス）** | 差分更新NN＋古典探索（AlphaZero とは別系統） | ❌ 未採用 |
